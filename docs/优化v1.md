你的分析很准，我会把它再收敛成一个工程上的结论：**现在不应该立刻去做大而全的 job_hunt / interview，而应该先把“可持续迭代简历”的闭环补齐。**

因为 OpenResume 现在的 Harness 底座已经像了：tool loop、context、state、trace、memory、slash、schema repair 都有了。真正卡用户体验的不是“还能不能生成”，而是：

> 用户说“把项目经历第二条改得更像 AI Agent 岗位”，系统能不能只改这一处、展示 diff、允许撤回、重新渲染并检查。

这个能力一旦有了，OpenResume 才从“生成器”变成“简历工作台”。

**我建议的继续优化顺序**
**第一优先级：Artifact 版本化 + diff + 回滚**

这是我认为下一个最应该做的点，比 job_hunt、mock interview 都更重要。

现在的问题是：`generate_resume_modules`、`render_latex`、`check_*` 会持续写 artifact，但没有统一的写入前快照，也没有 diff summary，更没有 rollback。Claude Code 类工具给人安全感的关键是“看得见它改了什么，也撤得回来”。

建议新增：

```text
resume_agent/artifacts/
  store.py          # artifact 路径、读写、快照
  diff.py           # 文本/JSON diff summary
  versioning.py     # versions/<timestamp>/...
```

新增工具：

```text
read_resume_artifact
snapshot_artifacts
diff_artifact
rollback_artifact
```

产物目录可以这样：

```text
project/
  versions/
    20260609_153012/
      latex/resume_modules.json
      drafts/resume.md
      metadata.json
```

这一步做完，后面的 `/revise`、`match_analysis`、`job_hunt` 才有安全底座。

**第二优先级：Section-level 增量修改**

你说的“最痛缺口”就是这个，我同意。

现在 `/revise` 只是 intent 和 workflow，真实能力还没有。下一步应该让它能改局部，而不是重新生成整份简历。

建议新增工具：

```text
read_resume_section
revise_resume_section
write_resume_section
```

支持 section：

```text
header
education
experience
projects
awards
skills
summary
```

用户输入：

```text
/revise 把项目经历写得更偏 AI Agent 和 RAG
/revise 只改实习经历，突出工具调用和自动化
/revise 精简奖项，让一页更稳
```

内部流程：

```text
读取 resume_modules.json
  -> 定位 section
  -> LLM 只生成该 section patch
  -> 合并到 resume_modules
  -> schema validate + repair
  -> snapshot old artifact
  -> 写入 new artifact
  -> render_latex
  -> check_truthfulness/check_ats
  -> 返回 diff summary
```

这会让 OpenResume 的核心体验立刻提升一档。

**第三优先级：match_analysis**

这个功能应该排在 job_hunt 前面。

原因很简单：就算你现在能搜到岗位，如果没有可靠的匹配分析，系统也不知道哪个岗位值得投、简历该怎么调。`match_analysis` 是岗位搜索、简历修改、面试准备三条线的共同中枢。

建议新增：

```text
resume_agent/tools/match_tools.py
```

工具：

```text
match_analysis
```

输入：

```text
project_dir
resume_modules.json
jd_analysis.md
spec_lock.json
```

输出：

```text
checks/match_report.json
```

报告结构：

```json
{
  "overall_score": 78,
  "keyword_coverage": {
    "covered": ["Python", "RAG", "AI Agent"],
    "missing": ["CI/CD", "Golang"],
    "coverage_rate": 0.68
  },
  "section_scores": {
    "experience": 82,
    "projects": 75,
    "skills": 70
  },
  "evidence_alignment": [
    {
      "jd_requirement": "Agent 框架设计",
      "resume_evidence": "OpenResume Agent 项目",
      "match": "strong"
    }
  ],
  "suggestions": [
    "项目经历中补充 Tool Calling 相关描述",
    "技能部分增加 CI/CD 或 Docker 证据"
  ]
}
```

这样 `/check` 就不只是 ATS 和真实性检查，而是真正能告诉用户“为什么这份简历还不够贴 JD”。

**第四优先级：Hooks 自动质量门禁（已完成轻量版）**

此前质量检查依赖模型记得调用工具。成熟 harness 不应该完全靠模型自觉。

现在已新增轻量 post-tool hooks 系统，没有做复杂框架，先解决“模型忘记检查”的体验问题。

例如：

```text
after generate_resume_modules:
  - check_truthfulness

after render_latex:
  - check_truthfulness
  - check_ats

after compile_pdf:
  - one-page/export_quality 强门禁
```

可以新增：

```text
resume_agent/engine/hooks.py
```

定义：

```python
POST_TOOL_HOOKS = {
    "generate_resume_modules": ["check_truthfulness"],
    "render_latex": ["check_truthfulness", "check_ats"],
    "revise_resume_section": ["check_truthfulness"],
    "revise_resume_from_match_report": ["check_truthfulness", "match_analysis"],
    "compile_pdf": [],
}
```

hook 已接入 `run_tool_plan` / `run_agent_loop`，结果会进入 trace、changed_files 和 warnings。这能让输出质量更稳定，也更像 Claude Code 那种“工具调用后系统自动补上下文和检查”的模式。

**第五优先级：PermissionPolicy，而不是只保留枚举（已完成基础版）**

此前 `ToolPermission` 是完整的，但主要是注册和过滤，没有真正做到“运行时审批/边界检查”。现在已新增基础版运行时 policy enforcement。

新增文件：

```text
resume_agent/tools/permissions.py
```

能力：

```text
allow READ
allow WORKSPACE_WRITE only under project_dir
deny DELETE by default
NETWORK needs explicit allow
EXPORT allowed
SENSITIVE_WRITE needs explicit runtime confirmation
```

短期不需要 UI approval，先做 policy enforcement：

```python
PermissionPolicy.check(tool, input_data, context)
```

尤其要防：

```text
写出 project_dir
读任意系统路径
network tool 默认无限制
subprocess script path 越界
```

当前实现已接入 `ToolRegistry.execute`：执行前拦截 NETWORK/DELETE/SENSITIVE_WRITE 和输入路径越界，执行后校验 outputs 仍在 project_dir 内。

**第六优先级：LaTeX repair + one-page gate（已完成基础版）**

`compile_pdf` 已从单次脚本调用增强为可靠导出闭环：

```text
render_pdf.py 编译
  -> 捕获失败和 resume.log 摘要
  -> LLM 修复 latex/resume.tex
  -> 重编译
  -> 检测 PDF 页数
  -> 超过 1 页时 LLM 精简 resume.tex
  -> 重编译
  -> 写 checks/export_quality.json
```

页数检测优先使用 PyMuPDF，回退到 `pdfinfo` 和 PDF 原始 page object 计数。若精简后仍超过 1 页，`compile_pdf` 会失败并写出明确 gate 报告。

**第七优先级：job_hunt MVP**

`/job_hunt` 的 MVP 现在已经落地到 harness 里了，但我仍然不建议一开始就做完整 MCP 或多平台登录态。招聘网站抓取很容易陷入反爬、登录、Cookie、分页、地区参数这些泥潭。

MVP 应该轻一点：

```text
tools/job_hunt_tools.py
  search_jobs
  select_job
```

先支持：

```text
用户输入岗位方向
  -> 从 profile.json 提取关键词
  -> 生成搜索 query
  -> 使用轻量搜索 backend
  -> 结构化为 jobs/jobs.jsonl
  -> 关键词匹配排序
  -> 用户选择某个岗位
  -> 写入 jobs/selected_job.json
  -> 写入 jd/jd_raw.md
```

不要一开始做：

```text
BOSS 直聘完整登录
LinkedIn 登录态
多平台专用解析器
定时监控
```

这些都属于 Phase 2 后段。

**第七优先级：模拟面试**

模拟面试很有价值，但应该排在 match_analysis 后面。

因为好的面试题依赖：

```text
profile facts
resume_modules
jd_analysis
match_report
```

没有 `match_report`，面试官就不知道该围绕哪些 gap 深挖。

建议等 `match_analysis` 出来后做：

```text
tools/interview_tools.py
  start_interview
  answer_interview_question
  end_interview
```

状态：

```text
ResumeStage.INTERVIEW
```

输出：

```text
checks/interview_report.json
```

**最应该先增加的功能**
如果只选一个，我建议先加：

```text
Artifact 版本化 + diff + rollback
```

如果选一个用户最能感知的，我建议先加：

```text
/revise 的 section-level 增量修改
```

但从工程顺序上，我会这样做：

```text
1. artifact snapshot/diff/rollback
2. read/write/revise section
3. match_analysis
4. hooks ✅
5. permission policy ✅
6. LaTeX repair loop ✅
7. one-page gate ✅
8. job_hunt MVP
9. mock_interview
```

**我的判断**
OpenResume 现在不缺“agent 骨架”，骨架已经挺像样了。下一步不要急着横向铺功能，而是先把主工作流变成一个稳定的“可编辑系统”。

一句话：

> 先做“改简历像改代码一样可见、可控、可撤回”，再做“找岗位”和“模拟面试”。

这个顺序最稳，也最像 Claude Code 的产品气质。
